What this is

Not a tutorial you read — a tracked plan you run like production. Every week is a GitHub issue → a branch → a build → a PR. "Done" means merged, and for infra weeks, Synced & Healthy in Argo CD. The repo doubles as a real infra-as-code stack that grows week by week.

01 — PLAN

A tracked roadmap

12 weekly milestones. Each is a self-contained folder with objectives, the build, and an end-of-week test.

02 — BUILD

Real infra-as-code

Manifests committed to the repo and synced to a learning EKS cluster by Argo CD, app-of-apps style.

03 — PROVE

Test before you tick

Every week ships a runnable test: auto checks on your artifacts + self-grade questions that prove you learned it.

The weekly loop

The same loop you'd want in prod: change is a PR, the cluster reflects git, drift self-heals.

# 1. open the week's issue, then branch
git checkout -b week-02-service-mesh

# 2. do the build — infra YAML under clusters/learning/, notes under notes/
# 3. prove it
bash weeks/week-02-service-mesh/test.sh

# 4. tick the ROADMAP box in the SAME PR, then open it
gh pr create        # closes the week's issue on merge
⚠ GPU cost discipline: rent ONE small GPU by the hour, scale EKS GPU nodes to zero when idle, tear it down after each session.

The 12 weeks

notes = a write-up you commit. infra = manifests Argo CD syncs. ★ = maps to real 2026 work goals.

Month 1Foundations & first inference deploy

Data-plane vs control-plane explained, a mesh canary running via Argo CD, a model served with vLLM.

Wk 01notes

Kubernetes networking refresher

Pod/Service model, CoreDNS, the VPC-CNI IP-exhaustion class of bug, L4 vs L7.

→ notes/week-01-k8s-networking.md
Wk 02infra

Service mesh: the mental model

Data plane vs control plane, Envoy sidecars vs istiod. Install Istio via Argo CD, prove mTLS.

→ manifests/istio/ + apps/istio.yaml
Wk 03infra

Istio resource model

VirtualService, DestinationRule, Gateway. An mTLS-enforced weighted canary with fault injection.

→ manifests/istio/ (canary)
Wk 04notesinfra

Inference mechanics + first serve

KV cache, continuous batching, PagedAttention. Serve a small model on one GPU, record TTFT & tokens/sec.

→ notes/week-04-inference-internals.md
Month 2Serving on Kubernetes + GPU + Argo

Model serves on EKS GPU with full observability, an Argo DAG designed not just debugged, GPU scheduling & quota understood.

Wk 05infra

GPU on Kubernetes

NVIDIA GPU Operator, MIG vs time-slicing, DRA. Schedule a GPU pod on EKS with Karpenter scale-to-zero.

→ manifests/gpu-operator/ + apps/
Wk 06infra★

Model serving on EKS (KServe)

InferenceService, cold-start realities. Serve behind KServe with a Grafana serving dashboard.

→ manifests/vllm/ + apps/vllm.yaml
Wk 07infra

Argo Workflows: troubleshoot → design

WorkflowTemplate, DAGs, params vs artifacts, retries & exit handlers. Rebuild a sync as a clean DAG.

→ manifests/workflows/
Wk 08infra

Job queueing + Argo ecosystem

Kueue admission & quota, Volcano awareness. Run a GPU job behind a quota; force a preemption.

→ manifests/kueue/ + apps/kueue.yaml
Month 3Mesh-routed, autoscaling inference (capstone)

An end-to-end GitOps-managed inference service on EKS you can explain, operate, and cost-justify.

Wk 09infra★

Inference at the gateway

Gateway API Inference Extension: InferencePool, model/LoRA-aware routing, token-aware rate limiting.

→ manifests/inference-gateway/ + apps/
Wk 10infra★

Autoscaling on the right signal

Why CPU HPA is wrong for inference. Scale vLLM on queue depth / TTFT via Prometheus adapter.

→ manifests/autoscaling/
Wk 11notes

Cost + reliability pass

GPU FinOps, Spot for inference, scale-to-zero. Inference SLOs, error budget, a PromQL alert.

→ notes/week-11-cost-slo.md
Wk 12infranotes

Capstone + write-up

The full stack wired together & Synced/Healthy, plus an architecture note + runbook.

→ notes/week-12-architecture.md
